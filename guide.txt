2. ARCHITECTURAL FLOW MATRIX (EACH FRAME LIFECYCLE)
   ========================================================================
   Every single frame, the system execution path follows a highly structured,
   deterministic sequence decoupled from presentation framerates:

[ XNA/FNA Game Loop Tick ]
│
▼
[ StateManager.Update() ] ──► Targets active State (GameplayState)
│
▼
[ GameplayState.Update() ]
│
▼
[ _ecsWorld.Progress(dt) ] ────► Dispatches Flecs native scheduler pipeline:
│
├─► [1. PreUpdate Stage]  ──► NetworkReceiverSystem.cs
│                             - Drains Steam P2P queue.
│                             - Updates or spawns remote shadow proxies.
│
├─► [2. OnUpdate Stage]   ──► MovementSystems.cs
│                             - Gathers local inputs to mutate Velocity.
│                             - Integrates Position += Velocity * TrueDelta.
│
└─► [3. PostUpdate Stage] ──► NetworkBroadcastSystem.cs
- Runs natively at a throttled 30Hz limit.
- Serializes states directly to raw memory.
- Dispatches P2P frames over Steam lines.

3. DETAILED SOURCE MODULE DOCUMENTATION
   ========================================================================

------------------------------------------------------------------------
MODULE: MyGame.csproj
------------------------------------------------------------------------
- TYPE: Configuration / MSBuild Script
- FUNCTION: Explicitly targets the .NET 10 runtime environment and links framework packages
  (Facepunch.Steamworks, Flecs.NET.Release, FontStashSharp, Aether.Physics2D, LiteDB).
- SOO OPTIMIZATIONS: Contains the `<AllowUnsafeBlocks>true</AllowUnsafeBlocks>` flag to
  enable performance-critical direct memory serialization. Injects `<DisableWinExeConsoleCheck>`
  to guarantee that text logged via `Console.WriteLine` reliably routes to an adjacent diagnostic
  cmd window in both Debug and optimized Release builds. Includes a wild-card deployment target macro
  (`..nalibsd\*.dll`) that automatically matches and moves required native binary dependencies
  (FNA3D.dll, SDL3.dll, etc.) right next to the finished executable when compiling via `dotnet publish`.

------------------------------------------------------------------------
MODULE: Engine/States/GameState.cs
------------------------------------------------------------------------
- TYPE: Core Base Class
- FUNCTION: Acts as the abstract interface blueprint for all presentation layers.
- SOC DESIGN: Enforces three distinct lifecycle methods across the engine: `LoadContent()`,
  `Update(GameTime)`, and `Draw(SpriteBatch)`. Crucially introduces `UnloadContent()`, a
  virtual destructor anchor that allows memory-intensive screens to drop hardware assets
  or unmanaged native pointer contexts cleanly whenever the active screen changes.

------------------------------------------------------------------------
MODULE: Engine/States/StateManager.cs
------------------------------------------------------------------------
- TYPE: Core Architecture Manager (Singleton)
- FUNCTION: Maintains the active execution window tree via a deferred operations list.
- SOC DESIGN: Solves state-switching memory safety. Rather than executing collection mutations
  mid-frame (which breaks call stacks), operations like `PushState`, `PopState`, and `ChangeState`
  are queued into a safe delegate collection (`pendingOperations`) and flushed at the boundary of
  the frame tick. Before any state is purged from the execution stack, `UnloadContent()` is called,
  preventing cross-state resource leaks.

------------------------------------------------------------------------
MODULE: Gameplay/Components/TransformComponents.cs
------------------------------------------------------------------------
- TYPE: Pure Data Structure (ECS Component Layout)
- FUNCTION: Contains `Position` and `Velocity` coordinate properties.
- SOC DESIGN: Strict data-only layout containing zero logic. Implemented as lightweight, value-typed
  C# structures. This isolates state variables completely from behavioral routines, allowing the
  Flecs engine to pack millions of these objects tightly inside continuous memory arrays for max CPU cache utilization.

------------------------------------------------------------------------
MODULE: Gameplay/Components/CharacterComponents.cs
------------------------------------------------------------------------
- TYPE: Pure Data Structure & ECS Tag Modifiers
- FUNCTION: Identifies state behavior types across generic entities.
- SOC DESIGN: Defines `CharacterClass` (containing a simple integer ID) alongside two completely empty structures:
  `LocalPlayerTag` and `RemotePlayerTag`. Because these structures contain no internal fields, Flecs handles them
  natively as zero-sized components (Tags). They carry zero memory footprint and act purely as fast filtering flags
  for specialized execution systems.

------------------------------------------------------------------------
MODULE: Gameplay/Components/NetworkComponents.cs
------------------------------------------------------------------------
- TYPE: Pure Data Structure (ECS Component Layout)
- FUNCTION: Stores authorization ownership signatures.
- SOC DESIGN: Wraps a Facepunch native `SteamId` value within a clean `NetworkOwner` construct. This separates
  identity credentials from physical game simulation components, providing systems an isolated path to determine
  who owns an entity across the network.

------------------------------------------------------------------------
MODULE: Gameplay/Networking/NetworkPackets.cs
------------------------------------------------------------------------
- TYPE: Binary Serialization Schema
- FUNCTION: Packs and unpacks network telemetry across P2P lines.
- SOC DESIGN: Highly optimized for performance. Uses a sequential struct layout layout (`[StructLayout(LayoutKind.Sequential, Pack = 1)]`)
  and unsafe raw memory pointers (`PlayerTransformPacket*`). By doing this, it completely bypasses the .NET garbage collector,
  avoiding the allocation of temporary `MemoryStream` or `BinaryWriter` objects. It copies the entire struct structure directly
  into or out of raw byte arrays using fixed memory addresses. This delivers zero-allocation, lighting-fast micro-serialization.

------------------------------------------------------------------------
MODULE: Gameplay/Networking/RemoteSessionManager.cs
------------------------------------------------------------------------
- TYPE: Network Tracking Service
- FUNCTION: Manages a mapping dictionary of active network connections to their respective proxy entities.
- SOC DESIGN: Keeps `GameplayState` clean by encapsulating state tracking into an independent network manager class.
  Exposes highly safe lookup tools using fast lookup routines (`TryGetValue`) to avoid multiple index scans per frame,
  and cleanly handles session invalidations (`ClearSessions()`) when disconnecting.

------------------------------------------------------------------------
MODULE: Gameplay/Prefabs/PlayerFactory.cs
------------------------------------------------------------------------
- TYPE: Creation Factory Blueprint
- FUNCTION: Assembles entities by combining required components into proper structural archetypes.
- SOC DESIGN: Isolates creation logic from both the systems and scenes. `CreateLocal()` composes a full player entity with
  authority flags, while `CreateRemote()` structures a remote shadow proxy entity mapped to a sender's network ID.
  This guarantees that entity composition remains consistent across the entire application.

------------------------------------------------------------------------
MODULE: Gameplay/Systems/MovementSystems.cs
--------------------------------─-──────────────────────────────────────
- TYPE: ECS Behavior System
- FUNCTION: Tracks keyboard actions and integrates delta-time movement equations.
- SOC DESIGN: Completely independent processor class. `InputSystem` reads raw hardware states and applies changes to
  entities tagged with `LocalPlayerTag`. `ApplyMovementSystem` runs universally on any entity possessing a `Position` and a `Velocity`
  component, modifying coordinates using the accurate, framework-synchronized `it.DeltaTime()`. This provides stable,
  framerate-independent physics behavior across 60Hz, 144Hz, or variable monitor refresh targets.

------------------------------------------------------------------------
MODULE: Gameplay/Systems/NetworkReceiverSystem.cs
------------------------------------------------------------------------
- TYPE: ECS Network Ingestion System
- FUNCTION: Drains Steam network buffers and updates the status of remote entities.
- SOC DESIGN: Configured to execute automatically during the `Ecs.PreUpdate` phase, meaning network changes are integrated
  right before physics movement processing runs. It reads raw bytes, decodes them via `PlayerTransformPacket.Deserialize`,
  and uses the `RemoteSessionManager` to either smoothly update existing shadow proxy locations or spin up a new network entity
  on the fly via the `PlayerFactory`.

------------------------------------------------------------------------
MODULE: Gameplay/Systems/NetworkBroadcastSystem.cs
------------------------------------------------------------------------
- TYPE: ECS Network Outbound System
- FUNCTION: Packages and broadcasts local transformations to all active lobby members.
- SOC DESIGN: A masterclass in performance separation. Runs during the `Ecs.PostUpdate` phase, guaranteeing that outbound packages
  reflect final, calculated movement transformations. Rather than flooding the network pipe every frame, it leverages Flecs'
  native `.Interval(1f / 30f)` scheduling modifier to automatically throttle updates to exactly 30 ticks per second. It performs
  this tracking internally, completely removing network timing clutter from the core game states.

------------------------------------------------------------------------
MODULE: GameStates/GameplayState.cs
------------------------------------------------------------------------
- TYPE: Scene Orchester / State Framework Component
- FUNCTION: Controls the graphics viewport and coordinates the main simulation life cycles.
- SOC DESIGN: Following this modular optimization, this file is beautifully lightweight. It contains no direct network code,
  no frame timing loops, and no raw packet parsing. It is responsible for exactly two core tasks:
    1. Bootstrapping: Creating the native Flecs simulation context, registering execution pipelines, and triggering factory blueprints.
    2. Rendering: Drawing active components back to the graphics device via a highly efficient, cached data query.
